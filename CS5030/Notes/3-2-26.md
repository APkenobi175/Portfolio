### MPI Wait Routines


#### Operating on distributed data

- You can use MPI collective operations to operate on distributed data.

- One dimensional data and linear process numbering
  - Operation between neighbouring indices: communication between neighbouring processes
  - Operation between all indices: communication between all processes

- Whats the problem with this approach?
  - there can be a block between sending and receiving processes.

- How do we plan around blocking?
  - We can use sendrecv, and this will make sure the exchange between those 2 ranks happens without any blocking.
  - We can have odds send to evens, and evens send to odds, and this will also avoid blocking.


#### Sendrecv

- instead of seperate send and recieve use `MPI_Sendrecv` to avoid blocking

```c
MPI_Sendrecv(sendbuf, sendcount, sendtype, dest, sendtag,
             recvbuf, recvcount, recvtype, source, recvtag,
             comm, status)
```

- `sendbuf`: starting address of send buffer
- `sendcount`: number of elements in send buffer
- `sendtype`: data type of send buffer elements
- `dest`: rank of destination process
- `sendtag`: send message tag
- `recvbuf`: starting address of receive buffer
- `recvcount`: number of elements in receive buffer
- `recvtype`: data type of receive buffer elements
- `source`: rank of source process
- `recvtag`: receive message tag
- `comm`: communicator
- `status`: status object (ignored if recvtag is `MPI_STATUS_IGNORE`)

#### Pairwise exchange

- Pairwise exchange: odd ranks send to even ranks, even ranks send to odd ranks

```c
MPI_Comm_rank(..., $procno);
```

#### Sending with irregular connections

- With irregular connections we can use graph operations.

- How do you approach this?
  - It is very hard to figure out a send/recieve sequence that does not deadlock or serialize.
  - Even if you manage that, you may have process idle time

- Instead
  - You can declare "This data needs to be sent" or "These messages are expected to be recieved"

- To do this we need a non-blocking send and recieve

  - // start non-blocking communication MPI_Isend(...); MPI_Irecv(...);
  - // wait for Isend/Irecv calls to finish in any order MPI_Wait(...);

- Using `MPI_Isend` and `MPI_Irecv` allows for non-blocking communication, which can help avoid deadlocks and reduce idle time in processes.
- `MPI_Wait` is used to ensure that all non-blocking operations have completed before proceeding with the next steps in the program.

#### Syntax for MPI_Isend and MPI_Irecv

```c
int MPI_Isend(sendbuf, sendcount, sendtype, dest, sendtag, comm, request);
int MPI_Irecv(recvbuf, recvcount, recvtype, source, recvtag, comm, request);
```

- `sendbuf`: starting address of send buffer
- `sendcount`: number of elements in send buffer
- `sendtype`: data type of send buffer elements
- `dest`: rank of destination process
- `sendtag`: send message tag
- `comm`: communicator
- `request`: request object `MPI_Request *request` used to track the non-blocking operation

- `recvbuf`: starting address of receive buffer
- `recvcount`: number of elements in receive buffer
- `recvtype`: data type of receive buffer elements
- `source`: rank of source process
- `recvtag`: receive message tag
- `comm`: communicator
- `request`: request object `MPI_Request *request` used to track the non-blocking operation

#### Comparison

- Blocking vs Non-blocking communication:
  - Blocking communication (e.g., `MPI_Send`, `MPI_Recv`) can lead to deadlocks if not carefully managed, especially in complex communication patterns.
  - Non-blocking communication (e.g., `MPI_Isend`, `MPI_Irecv`) allows for more flexible communication patterns and can help avoid deadlocks, but requires careful management of requests and synchronization.

- Buffer reuse
  - When a blocking call is made, the buffer can be reused immediately after the call returns.
  - When a non-blocking call is made, the buffer cannot be reused until the operation has completed, which can be checked using `MPI_Wait` or `MPI_Test`.

#### Latency Hiding

- Other reason for non blocking calls is you can overlap computation and communication, provided you have hardware support

- This is also known as latency hiding

- Exxample: Three-point combination operation
  - start communication for edge points
  - do local operations while communication is happening
  - wait for edge points from neighbor processes incorporate incoming data.

- Non blocking communication basically means we can do computations and communication at the same time, which can help hide the latency of communication and improve overall performance.

- That is AWESOME!

- Should we use regular send/recv over non-blocking send/recv?
  - It depends on the problem and the communication pattern. If the communication is simple and does not lead to deadlocks, blocking communication may be sufficient. However, for more complex communication patterns or when trying to optimize performance, non-blocking communication can be beneficial.
  - so rule of thumb is always try to use non-blocking communication no matter what because its easier.

- So one thread will be basically be like, I have this data can someone take care of it, and then the other thread will be like, I am waiting for this data, and then they will exchange that data without blocking each other.

#### Wait Routines

- Waits for MPI Isend or MPI Irecv to cmplete
  - request request (in), which is out parameter in MPI isend and MPI irecv
  - status status (out), which is ignored if `MPI_STATUS_IGNORE` is used

- Other routines include
  - `MPI_Waitall`: Waits for all specified requests to complete
  - `MPI_Waitany`: Waits for any one of the specified requests to complete
  - `MPI_Test`: Tests for the completion of a non-blocking operation without blocking the calling thread
  - `MPI_Testany`: Tests for the completion of any one of the specified requests without blocking the calling thread.

### MPI File I/O

- Data are not going to other ranks with MPI, but instead to a file on the disk.

- What is the problem with regular I/O in parallel?
  - you don't want all the processes to write to the same file at the same time, because that can lead to corruption of the file and performance issues.
  - Multiple processes read - no problem
  - Multiple processes write - big problem
  - Everyone writes to separate files - stress on file system and requires post processing to combine files

#### Opening a file - the API

```c
int MPI_File_open(MPI_Comm comm, const char *filename, int amode, MPI_Info info, MPI_File *fh);
```

- `comm`: communicator (e.g., `MPI_COMM_WORLD`)
- `filename`: name of the file to open
- `amode`: file access mode (e.g., `MPI_MODE_RDONLY`, `MPI_MODE_WRONLY`, `MPI_MODE_RDWR`, `MPI_MODE_CREATE`, `MPI_MODE_EXCL`)
- `info`: file information (e.g., hints for the file system)
- `fh`: file handle (output parameter)

- `MPI_File_open` is a collective call to opena  file
  - the collective is defined by the communicator
  - Typically you might use `MPI_COMM_WORLD` to have all processes in the communicator open the file together.

#### Closing a file - the API

- A number of access modes are supported for MPI file I/O, including:
  - `MPI_MODE_RDONLY`: Open the file for reading only
  - `MPI_MODE_WRONLY`: Open the file for writing only
  - `MPI_MODE_RDWR`: Open the file for both reading and writing
  - `MPI_MODE_CREATE`: Create the file if it does not exist
  - `MPI_MODE_EXCL`: Ensure that the file is created exclusively (i.e., fail if the file already exists)

```c
int MPI_File_close(MPI_File *fh);
```

- `MPI_File_close` is used to close a file that was previously opened with `MPI_File_open`.
- It takes a pointer to the file handle (`MPI_File *fh`) as an argument and returns an integer status code indicating success or failure of the operation. After calling `MPI_File_close`, the file handle is no longer valid and should not be used for further operations.

- This is similar to how you do it in regular C

#### Displacements, Elementary Datatypes, and Offsets

- The displacement of a position within a file is thenumber of bytes from the ebginning of the file
- an etype or elementary datatype is the basic unit of data that can be read or written to a file, such as `MPI_INT`, `MPI_FLOAT`, etc.
- An offset is a position in the file given the number of etypes from the beginning of the file

#### File Writing - the API

```c
int MPI_File_write_at(MPI_File fh, MPI_Offset offset, const void *buf, int count, MPI_Datatype datatype, MPI_Status *status);
```

- `MPI_File_write_at` is used to write data to a file at a specific offset.
- `fh`: file handle obtained from `MPI_File_open`
- `offset`: the position in the file where the data should be written, specified as the number of etypes from the beginning of the file
- `buf`: pointer to the buffer containing the data to be written
- `count`: number of elements to write
- `datatype`: the MPI datatype of the elements being written (e.g., `MPI_INT`, `MPI_FLOAT`, etc.)
- `status`: pointer to an `MPI_Status` object that will be filled with information about the write operation (e.g., number of bytes written, error status, etc.)

- This is just like sending data, you have a buffer, a count, and a datatype, but instead of sending it to another process, you are writing it to a file at a specific offset.

- **DON'T MESS THIS UP OR ONE THREAD MIGHT OVERWRITE ANOTHER THREAD'S DATA IN THE FILE**

- Calculating offset
  - If each process is writing a contiguous block of data, the offset can be calculated as `offset = rank * block_size`, where `rank` is the rank of the process and `block_size` is the size of the block of data each process is writing.
  - etype is the size of the data type being written (e.g., `sizeof(int)` for `MPI_INT`), so the offset can also be calculated as `offset = rank * block_size * etype` to account for the size of the data type.

##### Independant File Write

```c
offset = 12;
MPI_File_write_at(fh, offset, buf, count, datatype, &status);
```

#### File Views - the API

- A way to find a displacement portion in the output array of the file that you want to work on

```c
int MPI_File-set-view(MPI_File fh, MPI_Offset disp, MPI_Datatype etype, MPI_Datatype filetype, const char *datarep, MPI_Info info);
```
- `MPI_File_set_view` is used to define a file view, which specifies how data is accessed in the file.
- `fh`: file handle obtained from `MPI_File_open`
- `disp`: displacement from the beginning of the file to the start of the view, specified as the number of etypes
- `etype`: the elementary datatype that defines the basic unit of data access in the file (e.g., `MPI_INT`, `MPI_FLOAT`, etc.)
- `filetype`: the MPI datatype that defines the layout of the data in the file, which can be a derived datatype that describes a non-contiguous layout of data in the file
- `datarep`: a string that specifies the data representation (e.g., "native", "internal", "external32")
- `info`: file information (e.g., hints for the file system)

- The routine MPI_File_set_view is used to changed the veiw for a process 
- Each proess specifies an explicit disp (displacment, default is 0)

```c
disp = 8;
MPI_File_set_view(fh, disp, MPI_INT, MPI_INT, "native", MPI_INFO_NULL, &ierror);
offset = 1;
MPI_File_write_at(fh, offset, buf, count, datatype, &status);
```

- What this does is set the displacement to 8, which means that the view starts at the 8th integer in the file. Then when we call `MPI_File_write_at` with an offset of 1, it will write to the 9th integer in the file (8 + 1 = 9). This allows us to write to a specific location in the file without having to calculate the exact byte offset ourselves, as the file view takes care of that for us based on the specified displacement and datatypes.

```c
disp = 8;
Call MPI_File_set_view(fh, disp, MPI_INT, MPI_INT, "native", MPI_INFO_NULL, &ierror);
offset = 1;
Call MPI_File_write_at(fh, offset, buf, count, datatype, &status);
```

- This is an example of how to use file views to write to a specific location in the file. The actual values of `disp`, `offset`, `buf`, `count`, and `datatype` would depend on the specific application and the layout of the data in the file.

#### Reading and writing without offsets

- Routines are also provided so that you can write into a file without using explicit offsets, and instead the file pointer is maintained by MPI.
- in these cases you set the view prior to carrying out the write so that the view begins where you wish to write the data

```c
int MPI_File_write(MPI_File fh, const void *buf, int count, MPI_Datatype datatype, MPI_Status *status);
```

- This is `MPI_File_Write_at` without the at part. so it'll just write to the current position in the file.

- We have independent and collective writing of data
- For large data accesses collective data accesses can offer good performance improvements
- The collective routines have the same APIs as the independent routines except that the name of the routine ends in `_all`. For example, `MPI_File_write_at` becomes `MPI_File_write_at_all`.

- WHen you use collective, the runtime API will try to do better than just doing independent writes.

#### Writing Structured Data in Slabs

- One section of MPI user defined types in MPI 2 is specifically designed to improve data access for MPI IO
- MPI-2 introduced the idea of a subarray for data from a structured grid distributed on a set of processes
  - typically the domain decomposition defines a small cuboid inside a larger cuboid
  - possibly including halo regions

- By defining a subarray type we can use a collective call to write our data into one file.

- This helps you find those 3d regions in the file you want to write to

#### Using Subarray Types

```c
int MPI_Type_create_subarray(int ndims, const int *array_of_sizes, const int *array_of_subsizes, const int *array_of_starts, int order, MPI_Datatype oldtype, MPI_Datatype *newtype);
```

- `MPI_Type_create_subarray` is used to create a new MPI datatype that represents a subarray of a larger array.
- `ndims`: number of dimensions of the array
- `array_of_sizes`: array containing the size of each dimension of the full array
- `array_of_subsizes`: array containing the size of each dimension of the subarray
- `array_of_starts`: array containing the starting indices of the subarray within the full array
- `order`: the order of the array (e.g., `MPI_ORDER_C` for C-style row-major order, `MPI_ORDER_FORTRAN` for Fortran-style column-major order)
- `oldtype`: the MPI datatype of the elements in the array (e.g., `MPI_INT`, `MPI_FLOAT`, etc.)
- `newtype`: pointer to the new MPI datatype that will be created to represent the subarray

- We do not wanna use Fortran order because we are using C, so we will use `MPI_ORDER_C` for row-major order.

- In Fortran they are by column, where in c they are by row, so the order of the dimensions in the `array_of_sizes`, `array_of_subsizes`, and `array_of_starts` would be different depending on whether you are using C or Fortran order.


- Subarray Constructor Example

```c
Integer :: ranknum(4)

sizes = (/3, 4 /)
subsizes = (/1, 4 /)
starts = (/ wrank, 0 /)
Call MPI_Type_create_subarray(2, sizes, subsizes, starts, MPI_ORDER_C, MPI_INTEGER, subarray_type, ierror)
Call MPI_Type_commit(subarray_type, ierror)
ranknum(:) = wrank
discplacement = 0
Call MPI_File_set_view(fh, displacement, MPI_INTEGER, subarray_type, "native", MPI_INFO_NULL, ierror)
Call MPI_File_write_all(fh, ranknum, 4, MPI_INTEGER, status, ierror)
```

- So, in this example we build a subarray type that represents a 1x4 subarray of a 3x4 array. each process will write its rank number into the first row of the subarray, and the rest of the subarray will be filled with zeros. The file view is set to use the subarray type, so when we call `MPI_File_write_all`, each process will write its rank number into the correct position in the file according to the defined subarray layout.
- This allows us to write structured data into the file in a way that matches the layout of our data in memory, without having to manually calculate offsets for each process.
- When would we use this?
  - When we have a structured grid and we want to write out a specific portion of that grid to a file, especially in a parallel application where different processes are responsible for different portions of the grid. Using subarray types allows us to efficiently write out the data in a way that matches the structure of our data in memory, without having to manually calculate offsets for each process.
